# Relating Stuxnet to CERIAS Binary Hardening Research

My semester-long research project with CERIAS (Center for Education and Research in Information Assurance and Security — Purdue's cybersecurity research center), now a summer endeavor as I use my time back at home to get started on detailed documentation and benchmarking for Zephyr samples, was my first introduction to ICS/OT (Industrial Control Systems / Operational Technology) security. The research question I needed to confront is one of the most talked-about limiting factors in the subfield — how can security professionals implement robust security mechanisms on embedded hardware without betraying the reliability of the critical infrastructure that relies on it? More specifically, I was assigned Zephyr, which Dr. Tian called the "best open-source RTOS (Real-Time Operating System — a lightweight OS designed for resource-constrained embedded devices) the community has to offer", in order to examine binary hardening techniques for Cortex-M3 chips. With the hundreds of hours I have and will continue to put into this research question, it isn't surprising that I view embedded security through an entirely new lens. Every automobile, train, plane, piece of heavy machinery (which will be important here), or otherwise any important contributor to infrastructure here in the United States, is another piece of the attack surface vulnerable to exploitation that I see with fresh eyes.

When I looked for resources to deepen my understanding of ICS/OT security, Stuxnet was at the top of every result. It was one of many cases I enumerated in my search, so I resolved to put down a daily record of the topics I am actively reading and how they connect to my summer research.

Ultimately, not every ICS vulnerability is a result of any failure in the embedded space. Security has been an afterthought in embedded firmware for a generation now, as availability and reliability have taken priority in large, legacy, deterministic systems. Yet that isn't to say that the networks and security protocols surrounding the embedded attack surface are unfamiliar to security professionals. Usually, from what I've read about Stuxnet and other attacks on physical infrastructure, although embedded security — or the lack thereof — enables the exploitation of a system, it is often not the direct cause of it.

In the Stuxnet attack, unauthorized access to PLC (Programmable Logic Controller — the embedded computer directly controlling industrial machinery like centrifuges) configurations was gained through infected USB drives, four Windows zero-day vulnerabilities, and the forging of digital signatures. There is little doubt that significant covert intelligence gathering contributed as well. The point is that embedded systems are the last line of defense on the court, and they have their hands down. The hard part of the Stuxnet attack flow was getting to the point where infected systems could inject malicious configurations into the PLC — the easy part was doing whatever you want once you were already in.

So why do embedded systems lack the robust mechanisms so present in other areas of security? From what I have learned in my research, it comes down to three reasons: it is expensive, the performance constraint tradeoffs aren't worth the added security to most engineers, and OT as a whole persists with old infrastructure because of the downtime required for large system changes.

Embedded security is expensive, and the reason ties directly into the inherent performance constraints of constrained chips like the Cortex-M3 I work on. Embedded devices operate with low clock speeds and limited RAM. As a result, engineers need specialized security solutions to achieve any meaningful level of protection. In my research, I encountered the memory constraints of these chips firsthand and learned a great deal about the difference between physical and virtual memory. On embedded devices, there is typically a fixed amount of physical memory assigned at boot — there is no room for the overhead that virtual memory mapping and the extra cycles it requires. I've read about solutions like MPUs (Memory Protection Units — hardware components that enforce memory access boundaries between regions) and Dr. Tian's CRT-C, a memory-safe approach to vulnerable C language features. Discovering these solutions is expensive in the first place, yet the biggest cost comes when implementing them into old, proven systems. For the heavy machinery that operates with constrained devices, reliability and uptime are the highest priorities. Even a second of delay on a manufacturing line results in money lost. Even a second of delay on centrifuge operation could set back an enrichment program by days.

After researching Stuxnet's attack flow — specifically how attackers operated once they had already compromised the system — I identified a few ways the PLC configuration mechanism could have been more resistant to this class of attack.

1. **Code signing on PLC uploads** — Stuxnet injected code blocks with no authentication whatsoever. Requiring cryptographically signed code blocks during the upload process would have rejected the malware's attempt to alter centrifuge behavior. Code signing carries only minimal runtime overhead since verification occurs at upload time, not during execution.

2. **HMAC on Step 7 communications** — In Stuxnet's case, the engineer's workstation sent legitimate PLC programs through Siemens' proprietary S7 protocol to the controller. The malware's DLL (Dynamic Link Library — a reusable code module that Windows applications load at runtime) sat inside the Step 7 process and intercepted the data stream mid-transfer to append malicious code blocks. The PLC received no indication of on-path tampering. An HMAC (Hash-based Message Authentication Code — a cryptographic checksum that verifies both the integrity and authenticity of a message) on the communication channel would have detected modifications to the upload stream before anything malicious reached the controller.

3. **Hardware-enforced VFD operating bounds** — This one is less about the Step 7 process and more about a safety mechanism on the centrifuge itself. A bounds checker at the VFD (Variable Frequency Drive — the electronic controller regulating centrifuge motor spin speed) level, independent of PLC logic, would refuse any command outside safe RPM (Revolutions Per Minute) parameters. This requires no cryptography and near-zero compute overhead, and it would have prevented the physical destruction of the centrifuges entirely.

Overall, Stuxnet is interesting not only for what it reveals about cybersecurity, but also in the broader geopolitical context of uranium enrichment and ongoing US-Iran tensions. I'll be returning to both threads throughout this summer's reading.
